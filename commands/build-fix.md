# Build and Fix

ビルドエラーや型エラーを最小限・安全な変更で段階的に修正します。

## ステップ 1: ビルドシステムの検出

プロジェクトのビルドツールを特定してビルドを実行します：

| 判断材料 | ビルドコマンド |
|-----------|---------------|
| `build` スクリプトを持つ `package.json` | `npm run build` または `pnpm build` |
| `tsconfig.json`（TypeScript のみ） | `npx tsc --noEmit` |
| `Cargo.toml` | `cargo build 2>&1` |
| `pom.xml` | `mvn compile` |
| `build.gradle` | `./gradlew compileJava` |
| `go.mod` | `go build ./...` |
| `pyproject.toml` | `python -m py_compile` または `mypy .` |

## ステップ 2: エラーの解析とグループ化

1. ビルドコマンドを実行して stderr をキャプチャする
2. ファイルパスごとにエラーをグループ化する
3. 依存関係の順番でソートする（ロジックエラーの前にインポート・型を修正する）
4. 進捗追跡のためにエラーの総数をカウントする

## ステップ 3: 修正ループ（1つずつ）

各エラーについて：

1. **ファイルを読む** — Read ツールでエラー周辺（前後10行）のコンテキストを確認する
2. **診断** — 根本原因を特定する（インポート漏れ、型の誤り、構文エラー）
3. **最小限の修正** — Edit ツールでエラーを解消する最小の変更を行う
4. **ビルドを再実行** — エラーが消えて新たなエラーが発生していないことを確認する
5. **次へ** — 残りのエラーに対して続ける

## ステップ 4: 停止基準

以下の場合はユーザーに確認を求めて停止する：
- 修正によって**解消するより多くのエラーが発生する**場合
- **同じエラーが 3 回試みても解消しない**場合（より深い問題の可能性）
- 修正に**アーキテクチャ変更が必要**な場合（ビルド修正の範囲外）
- ビルドエラーが**不足している依存関係**に起因する場合（`npm install`、`cargo add` 等が必要）

## ステップ 5: サマリー

結果を表示する：
- 修正したエラー（ファイルパス付き）
- 残っているエラー（あれば）
- 新たに発生したエラー（ゼロであるべき）
- 未解決の問題に対する次のステップの提案

## 回復戦略

| 状況 | アクション |
|-----------|--------|
| モジュール・インポートの欠如 | パッケージがインストールされているか確認し、インストールコマンドを提案する |
| 型の不一致 | 両方の型定義を読み、より狭い型に修正する |
| 循環依存 | インポートグラフでサイクルを特定し、抽出を提案する |
| バージョン競合 | `package.json` / `Cargo.toml` のバージョン制約を確認する |
| ビルドツールの設定ミス | 設定ファイルを読み、動作するデフォルト設定と比較する |

安全のために 1 エラーずつ修正する。リファクタリングより最小差分を優先する。
