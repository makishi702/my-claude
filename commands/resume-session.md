# Resume Session コマンド

最後に保存されたセッション状態を読み込み、作業を始める前に完全に状況を把握します。
このコマンドは `/save-session` の対となるものです。

## 使いどころ

- 前日の作業を続けるために新しいセッションを開始するとき
- コンテキスト上限のため新しいセッションを開始した後
- 別のソースからのセッションファイルを引き継ぐとき（ファイルパスを指定する）
- セッションファイルがあり、作業を進める前に Claude に完全に吸収させたいとき

## 使い方

```
/resume-session                                                      # ~/.claude/sessions/ の最新ファイルを読み込む
/resume-session 2024-01-15                                           # その日付の最新セッションを読み込む
/resume-session ~/.claude/sessions/2024-01-15-session.tmp           # レガシー形式の特定ファイルを読み込む
/resume-session ~/.claude/sessions/2024-01-15-abc123de-session.tmp  # 現在の short-id セッションファイルを読み込む
```

## プロセス

### ステップ 1: セッションファイルを見つける

引数が指定されていない場合：

1. `~/.claude/sessions/` を確認する
2. 最も最近変更された `*-session.tmp` ファイルを選ぶ
3. フォルダーが存在しないまたはマッチするファイルがない場合、ユーザーに伝える：
   ```
   ~/.claude/sessions/ にセッションファイルが見つかりません
   セッション終了時に /save-session を実行して作成してください。
   ```
   その後、停止する。

引数が指定されている場合：

- 日付（`YYYY-MM-DD`）のように見える場合、`~/.claude/sessions/` で `YYYY-MM-DD-session.tmp`（レガシー形式）または `YYYY-MM-DD-<shortid>-session.tmp`（現在の形式）にマッチするファイルを検索し、その日付の最も最近変更されたバリアントを読み込む
- ファイルパスのように見える場合、そのファイルを直接読む
- 見つからない場合は明確に報告して停止する

### ステップ 2: セッションファイル全体を読む

ファイル全体を読む。まだ要約しない。

### ステップ 3: 理解を確認する

以下の正確な形式で構造化されたブリーフィングを返す：

```
セッション読み込み完了: [ファイルへの実際のパス]
════════════════════════════════════════════════

プロジェクト: [ファイルのプロジェクト名・トピック]

構築しているもの:
[自分の言葉で 2〜3 文のサマリー]

現在の状態:
✅ 動作確認済み: [確認済み項目数]
🔄 進行中: [進行中のファイルの一覧]
🗒️ 未着手: [計画済みだが未変更のものの一覧]

再試行しないこと:
[失敗した全アプローチとその理由の一覧 — これは重要]

未解決の問題・ブロッカー:
[ブロッカーまたは未回答の質問の一覧]

次のステップ:
[ファイルに定義されている場合は正確な次のステップ]
[定義されていない場合: 「次のステップ未定 — 開始前に『まだ試していないこと』を一緒に確認することを推奨」]

════════════════════════════════════════════════
再開の準備ができました。どうしますか？
```

### ステップ 4: ユーザーを待つ

自動的に作業を開始しないこと。ファイルに一切触れないこと。ユーザーが次に何をするかを言うのを待つ。

セッションファイルに次のステップが明確に定義されており、ユーザーが「続けて」「yes」などと言った場合 — その正確な次のステップで進める。

次のステップが定義されていない場合 — ユーザーにどこから始めるかを確認し、必要に応じて「まだ試していないこと」セクションからアプローチを提案する。

## エッジケース

**同じ日付に複数のセッションがある場合**（`2024-01-15-session.tmp`、`2024-01-15-abc123de-session.tmp`）：
レガシーの no-id 形式か現在の short-id 形式かにかかわらず、その日付のマッチするファイルの中で最も最近変更されたものを読み込む。

**セッションファイルが参照するファイルが存在しない場合：**
ブリーフィング時に明記する — 「⚠️ セッションで参照されている `path/to/file.ts` がディスク上に見つかりません。」

**セッションファイルが 7 日以上前のもの：**
ギャップを明記する — 「⚠️ このセッションは N 日前のものです（閾値: 7 日）。変更されている可能性があります。」— その後、通常通り進める。

**ユーザーがファイルパスを直接指定した場合（例：チームメイトから転送された）：**
同じブリーフィングプロセスに従って読む — ソースにかかわらず形式は同じ。

**セッションファイルが空または壊れている場合：**
報告する: 「セッションファイルは見つかりましたが、空または読み取り不能です。/save-session で新しいものを作成する必要があるかもしれません。」

## 注意事項

- 読み込む際にセッションファイルを変更しないこと — それは読み取り専用の歴史的記録
- ブリーフィングの形式は固定 — セクションが空でも省略しないこと
- 「再試行しないこと」は常に表示すること（「なし」であっても） — 見落とすには重要すぎる
- 再開後、ユーザーはその日の終わりに再び `/save-session` を実行して新しい日付付きファイルを作成したいかもしれない
