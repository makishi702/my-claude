# ドキュメントの更新

ソースとなるファイルから生成することで、ドキュメントをコードベースと同期させる。

## ステップ 1: 情報源（Source of Truth）の特定

| ソース | 生成されるドキュメント |
|--------|-----------|
| `package.json` のスクリプト | 利用可能なコマンドのリファレンス |
| `.env.example` | 環境変数のドキュメント |
| `openapi.yaml` / ルートファイル | APIエンドポイントリファレンス |
| ソースコードのエクスポート | パブリックAPIドキュメント |
| `Dockerfile` / `docker-compose.yml` | インフラセットアップドキュメント |

## ステップ 2: スクリプトリファレンスの生成

1. `package.json`（または `Makefile`、`Cargo.toml`、`pyproject.toml`）を読み込む
2. すべてのスクリプト・コマンドとその説明を抽出する
3. リファレンステーブルを生成する：

```markdown
| コマンド | 説明 |
|---------|-------------|
| `npm run dev` | ホットリロード付き開発サーバーの起動 |
| `npm run build` | 型チェック付きプロダクションビルド |
| `npm test` | カバレッジ付きテストスイートの実行 |
```

## ステップ 3: 環境変数ドキュメントの生成

1. `.env.example`（または `.env.template`、`.env.sample`）を読み込む
2. すべての変数とその用途を抽出する
3. 必須・任意に分類する
4. 期待されるフォーマットと有効な値をドキュメント化する

```markdown
| 変数 | 必須 | 説明 | 例 |
|----------|----------|-------------|---------|
| `DATABASE_URL` | Yes | PostgreSQL 接続文字列 | `postgres://user:pass@host:5432/db` |
| `LOG_LEVEL` | No | ログの詳細度（デフォルト: info） | `debug`, `info`, `warn`, `error` |
```

## ステップ 4: コントリビューションガイドの更新

`docs/CONTRIBUTING.md` を生成・更新し、以下を含める：
- 開発環境のセットアップ（前提条件、インストール手順）
- 利用可能なスクリプトとその用途
- テスト手順（実行方法、新規テストの書き方）
- コードスタイルの強制（linter、フォーマッター、pre-commit hook）
- PR 提出チェックリスト

## ステップ 5: ランブックの更新

`docs/RUNBOOK.md` を生成・更新し、以下を含める：
- デプロイ手順（ステップバイステップ）
- ヘルスチェックエンドポイントとモニタリング
- よくある問題とその対処法
- ロールバック手順
- アラートとエスカレーションパス

## ステップ 6: 陳腐化チェック

1. 90日以上更新されていないドキュメントファイルを検出する
2. 最近のソースコード変更と照合する
3. 手動レビューが必要な可能性のある古いドキュメントにフラグを立てる

## ステップ 7: サマリーの表示

```
Documentation Update
──────────────────────────────
Updated:  docs/CONTRIBUTING.md (スクリプトテーブル)
Updated:  docs/ENV.md (3 new variables)
Flagged:  docs/DEPLOY.md (142 days stale)
Skipped:  docs/API.md (no changes detected)
──────────────────────────────
```

## ルール

- **単一の情報源**: 常にコードから生成し、生成されたセクションを手動編集しない
- **手動セクションの保護**: 生成されたセクションのみ更新し、手書きの文章はそのまま残す
- **生成コンテンツのマーク**: 生成されたセクションは `<!-- AUTO-GENERATED -->` マーカーで囲む
- **指示なしにドキュメントを作成しない**: コマンドで明示的に要求された場合のみ新規ドキュメントファイルを作成する
