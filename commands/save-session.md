# Save Session コマンド

このセッションで起きたことすべて（構築したもの、成功したもの、失敗したもの、残っているもの）をキャプチャし、次のセッションがここで終わったところから正確に再開できるように、日付付きファイルに書き込みます。

## 使いどころ

- Claude Code を閉じる前の作業セッションの終わり
- コンテキスト上限に達する前（まずこれを実行してから新しいセッションを開始する）
- 後で参照したい複雑な問題を解決した後
- 将来のセッションにコンテキストを引き継ぎたいとき

## プロセス

### ステップ 1: コンテキストの収集

ファイルを書き込む前に以下を収集する：

- このセッションで変更された全ファイルを読む（git diff または会話から確認する）
- 議論・試みた内容・決定事項を振り返る
- 発生したエラーとその解決方法（または未解決のまま）を記録する
- 関連する場合は現在のテスト・ビルドの状態を確認する

### ステップ 2: sessions フォルダーが存在しない場合は作成する

ユーザーの Claude ホームディレクトリに標準の sessions フォルダーを作成する：

```bash
mkdir -p ~/.claude/sessions
```

### ステップ 3: セッションファイルを書き込む

`~/.claude/sessions/YYYY-MM-DD-<short-id>-session.tmp` を作成する。今日の実際の日付と `session-manager.js` の `SESSION_FILENAME_REGEX` で強制されるルールを満たす short-id を使用する：

- 使用可能な文字: 小文字 `a-z`、数字 `0-9`、ハイフン `-`
- 最小長: 8 文字
- 大文字・アンダースコア・スペースは不可

有効な例: `abc123de`、`a1b2c3d4`、`frontend-worktree-1`
無効な例: `ABC123de`（大文字）、`short`（8 文字未満）、`test_id1`（アンダースコア）

完全な有効ファイル名の例: `2024-01-15-abc123de-session.tmp`

レガシーのファイル名 `YYYY-MM-DD-session.tmp` も有効だが、新しいセッションファイルは同日の衝突を避けるために short-id 形式を優先する。

### ステップ 4: 以下の全セクションをファイルに記載する

全セクションを正直に書く。セクションを省略しないこと — 本当に内容がない場合は「まだなし」や「N/A」と書く。不完全なファイルは正直に空のセクションを書くより悪い。

### ステップ 5: ファイルをユーザーに表示する

書き込んだ後、完全な内容を表示して以下を確認する：

```
セッションを保存しました: [セッションファイルへの実際のパス]

正確に見えますか？閉じる前に修正や追加はありますか？
```

確認を待つ。要望があれば編集する。

## 構築しているもの

Next.js アプリ用のユーザー認証システム。ユーザーはメール・パスワードで登録し、localStorage ではなく httpOnly cookie に保存された JWT を受け取り、保護されたルートはミドルウェアを介して有効なトークンを確認する。目標はトークンを JavaScript に公開せずにブラウザリフレッシュをまたいでセッションを永続化すること。

---

## うまくいったこと（証拠付き）

- **`/api/auth/register` エンドポイント** — 確認方法: Postman POST がユーザーオブジェクトで 200 を返す、Supabase ダッシュボードで行が見える、bcrypt ハッシュが正しく保存されている
- **`lib/auth.ts` の JWT 生成** — 確認方法: ユニットテストが合格（`npm test -- auth.test.ts`）、jwt.io でデコードしたトークンが正しいペイロードを示している
- **パスワードハッシュ** — 確認方法: テストで `bcrypt.compare()` が true を返す

---

## うまくいかなかったこと（理由付き）

- **Next-Auth ライブラリ** — 失敗した理由: カスタム Prisma アダプターと競合し、全リクエストで「Cannot use adapter with credentials provider in this configuration」がスローされた。デバッグの価値なし — 我々のセットアップには制約が強すぎる。
- **localStorage での JWT 保存** — 失敗した理由: SSR レンダリングは localStorage が利用可能になる前に発生し、全ページ読み込みで React ハイドレーションのミスマッチエラーが発生した。このアプローチは Next.js SSR と根本的に非互換。

---

## まだ試していないこと

- ログインルートのレスポンスで JWT を httpOnly cookie として保存する（最有力の解決策）
- サーバーコンポーネントでトークンを読むために `next/headers` の `cookies()` を使用する
- cookie の存在確認によってルートを保護する middleware.ts を書く

## 行われた決定事項

- **localStorage より httpOnly cookie** — 理由: XSS によるトークン窃取を防ぐ、SSR で動作する
- **Next-Auth よりカスタム認証** — 理由: Next-Auth が我々の Prisma セットアップと競合し、対処する価値なし

## 注意事項

- 各セッションは独自のファイルを持つ — 以前のセッションのファイルに追記しない
- 「うまくいかなかったこと」セクションが最も重要 — このセクションなしでは将来のセッションが失敗したアプローチを再試行する
- ユーザーがセッション中（終了時だけでなく）の保存を求めた場合、現時点で判明している内容を保存し、進行中の項目を明確にマークする
- このファイルは次のセッションの開始時に `/resume-session` で Claude に読まれることを想定している
- 標準のグローバルセッションストアを使用する: `~/.claude/sessions/`
- 新しいセッションファイルには short-id 形式（`YYYY-MM-DD-<short-id>-session.tmp`）を優先する
